# リファクタリング調査レポート（2026-07-24）

> 依頼: 「バグが多発して、全体的によくない気がするので、リファクタリング調査をして。実装はしないで、調査に徹して」
> 本レポートは調査のみ。コードは一切変更していない。

## 0. 調査の範囲と方法

- **macOS**: 実機で実施。実バイナリでのトーン再生録音（AutoGain × 手動ミキサー × source の 9 通り）、デバイス切替の再現実験、本番 vault の実録音ファイルの解析、`sysrec.swift` 全構造の読解。
- **Windows**: 実機が無いためコード読解＋レンダラ実験（macOS 上の Obsidian は Windows 版と同じ Chromium 142 なので、Web Audio / MediaRecorder / WebCodecs の挙動の強い代理になる）。**実機でしか確かめられない項目は §4.3 に明示**した。
- 対象: `src/` 全域（約 8,500 行）、`native/sysrec/sysrec.swift`（1,478 行）、`test/e2e.mjs`、直近の障害ログ・録音ファイル。

## 1. バグ多発の正体 — 直近 7 件の構造要因マップ

| バグ | プラットフォーム | 直接原因 | **構造要因** |
|---|---|---|---|
| #1 マイク既定以外で録音不可 | macOS | 非既定デバイスへの IOProc がハング | **B** デバイスライフサイクル |
| #4 AutoGain オフが実質無音 | macOS | 正規化を手動時に外していた | **A** DSP 仕様の分散（掛け分け条件が場所ごとに違う） |
| #4 Windows が無音/小さい | Windows | AGC が no-op・ノード GC・suspended | **A**＋**C** 実機検証ゼロのまま出荷 |
| 旧 sysrec で録音 0 バイト（7/24 10:22） | macOS | main.js だけ更新されバイナリ据え置き | **G** 配布 4 点セットの不可分性が未管理だった |
| AutoGain オンの方が 18dB 小さい（7/24） | macOS | AGC ゲート未満は無補正＋normalize スキップ | **A** AGC と normalize の役割の混同 |
| 出力音量で録音レベルが変わる（7/24） | Windows | ループバックは音量つまみの後ろ | **D** プラットフォーム差の把握漏れ（macOS の常識を敷衍） |
| **システム音が途中で 17 秒で停止（7/24 12:39・未修正）** | macOS | タップの集約デバイスが開始時の既定出力に固定 | **B** デバイスライフサイクル |

**結論: 7 件中 6 件が「A: DSP 仕様の分散」「B: デバイスライフサイクル」「C: 検証ギャップ」の 3 要因に集約される。** UI や状態機械（start/stop/sweep/remix）起因のバグはゼロ。Phase 0–3 で作り込んだ状態機械・堅牢性層は E2E に守られて安定しており、**壊れ続けているのは「音声パイプライン」と「デバイスとの接点」**である。

## 2. 未修正の既知バグ（最優先・リファクタ以前）

### 2.1 macOS: 出力デバイス切替でシステム音の取り込みが死ぬ（再現済み）

- 症状: 12:39 の録音で mic は 51.2 秒、system は **17.2 秒で停止**（`system=412312 / mic=1228776` フレーム）。ファイルは正常に見えるが相手の声が途中から消えている。
- 原因: [sysrec.swift:770-784](native/sysrec/sysrec.swift#L770-L784) — タップ駆動用の集約デバイスを**録音開始時点の既定出力デバイス UID に固定**して作り、`kAudioHardwarePropertyDefaultOutputDevice` の変更を監視していない。Bluetooth イヤホン（EDIFIER EvoBuds）の接続/切断・出力先切替で以降のシステム音が取れない。
- 再現: 録音中に既定出力を切り替える実験で確認。切替後の再生音は録音に入らない（フレームは刻み続けるが無音）。
- 設計書に「残（実機・未検証）: 出力デバイス切替」と**既知の未検証項目として記載されていたものが実害化**した。
- 修正方針（実装は未着手）: 既定出力の変更リスナーを張り、変更時に集約デバイスを作り直して IOProc を張り替える。タップ自体は生かせるので録音は途切れず継続できる見込み。CLI 契約は変わらない（abi 据え置き）。
- 併記すべき観測: 現環境（EvoBuds が既定出力・BT 接続表示あり）で**開始直後から全区間無音**になるケースも観測した。BT の A2DP アイドル状態が絡む可能性があり、リスナー修正だけで足りるかは実機で追試が要る。

### 2.2 Windows: 同種の疑問が未検証

Windows のシステム音は Electron の `getDisplayMedia` + `audio:'loopback'`（WASAPI ループバック）。**録音中に既定出力デバイスを切り替えたときループバックが追従するのか、トラックが `ended` になるのか、無音を流し続けるのかは実機未確認**（§4.3）。Windows は track `ended` → 自動 finalize の fail-safe があるため「ファイルが半端に残る」方向には倒れにくいが、「黙って無音」の可能性は否定できない。

## 3. 構造課題（優先度順）

### A. DSP 仕様の多重実装 — バグ製造機の本体

レベル処理の定数・ロジックが **Swift と TypeScript に手動同期の二重実装**になっている:

| 処理 | macOS 実装 | Windows 実装 | 同期手段 |
|---|---|---|---|
| AGC（-20dBFS 目標/ゲート -42/+12〜-18dB/上げ遅く下げ速く/再アーム） | `sysrec.swift` AGCProcessor（156 行〜） | `agc.ts` nextAgcState | **人力**（コメントのみ） |
| 仕上げ正規化（-16dBFS/±18dB/ゲート空振り救済/無音素通し） | `sysrec.swift` loudnessGain（274 行〜） | `agc.ts` nextNormalizerState | **人力** |
| リミッター（-1dBFS 常時） | StreamingLimiter ＋ applyLookaheadLimiter | DynamicsCompressor 設定 | **人力** |
| ノイズゲート | NoiseGate（384 行〜） | **無い**（Chromium の noiseSuppression に委任） | — |
| 手動ゲイン | ManualGain（control ファイル経由） | GainNode 直接 | **人力** |

問題の実例: 「AGC はゲート未満に効かない」という仕様理解が normalize の掛け分け判断（stop.ts）に反映されておらず「AutoGain オンの方が小さい」が生まれた。**仕様が 1 箇所に書かれていないため、変更のたびに片側だけ直る／条件分岐の前提がずれる**。

さらに適用順・適用場所も分散している: mix の `--normalize on` は [mix.ts:39](src/recorder/mix.ts#L39)、single の normalize は [stop.ts](src/recorder/stop.ts)、rescue の normalize は [mix.ts の mixOrRescue](src/recorder/mix.ts#L159-L165)、Windows は取り込み時 normGain。「どの経路でどの補正が掛かるか」の一覧表がコード上のどこにもない（今回この調査で初めて表になった）。

### B. デバイスライフサイクルの無視（macOS 側が特に弱い）

| イベント | macOS (sysrec) | Windows (webCapture) |
|---|---|---|
| 既定出力デバイスの変更 | **追従しない → 無音（§2.1）** | 未検証（§4.3） |
| デバイス切断（BT 等） | **検知しない → 無音のまま録り続ける** | track `ended` → **自動 finalize（fail-safe）** |
| 録音中の AudioContext suspended | 該当なし | statechange で自動 resume 済み |
| 無音の継続 | 検知しない | 5 秒無音ウォッチ → Notice 警告 |
| マイクデバイスの消失 | 検知しない | `ended` → finalize |

皮肉なことに **後発の Windows 実装の方が防御的**で、macOS 側（信頼していた sysrec）にこそ穴が残っている。macOS に足りないもの: ①既定出力変更リスナー ②デバイス消失検知 ③無音ウォッチ（sysrec は level を書いているのでプラグイン側で監視可能な材料は既にある）。

### C. 検証ギャップ — E2E は「状態機械」しか見ていない

現状の E2E 68 件の内訳: fake-binary による状態機械検証（start/stop/mix/rescue/sweep/restore/watch）＋ DSP 純関数の数値検証。**すべてプロセス管理とロジックの検証であり、音声そのものを 1 サンプルも検証していない**。

検証されていない層（今回のバグは全部ここで起きた）:
1. **実バイナリの CLI 契約**（旧バイナリ事故: fake は常に最新契約を演じるので検出不能）→ 版数ハンドシェイクで部分対処済み
2. **実音声のレベル**（無音・小さい・逆転）→ 今回手動でやった「トーン再生 → 録音 → ffmpeg で dBFS 計測」のマトリクスは全て自動化可能（macOS ローカル限定）
3. **デバイス変化**（切替・切断）→ 今回作った `switch-out.swift` の要領でスクリプト化可能
4. **Windows 実機**（AGC・正規化・ループバック挙動・mp4 対応）→ CI に Windows ランナーを足すか、`dev/win-audio-diag.js` を「ユーザー実行 1 コマンド＋結果 JSON 貼り付け」まで整備するか

### D. プラットフォーム抽象の欠如

`process.platform` 分岐が 13 箇所・6 ファイルに散在。start/stop/レベル取得/ミキサー反映/復元がそれぞれ別名の対（start↔startWeb、stop↔stopWeb、LevelPoller↔getSourceLevels、control ファイル↔GainNode 直接）になっており、**main.ts が接着剤としてすべての分岐を持つ**（854 行・メソッド 25 個の一因）。

「録音バックエンド」インターフェース（start/stop/levels/setGains/watch）を 1 枚定義して macOS 実装（sysrec 駆動）と Windows 実装（WebRecorder）を対にすれば、main.ts の分岐は消え、新規パス追加時に「もう片方を忘れる」構造事故（#4 の AGC 未配線がまさにこれ）を型で防げる。

### E. `sysrec.swift` モノリス（1,478 行）

1 ファイルに 11 クラス＋10 関数: 引数解析／AGC／手動ゲイン／リミッター2種／ノイズゲート／正規化／WriterBox／フォーマット変換／CoreAudio デバイスユーティリティ／TapCapturer／MicCapturer／Capture 統括／mix／normalize／エントリ。前回調査（2026-07-05）でも分割候補に挙がり「検証済みバイナリを churn しない」ため保留された。**§2.1 の修正でどのみち TapCapturer を触るので、そのタイミングで DeviceMonitor 周りだけ切り出すのが低リスク**。ビルドは swiftc 直呼びなので複数ファイル化は build.sh の引数追加のみ。

### F. `main.ts` の神クラス傾向（854 行・25 メソッド）

責務: ライフサイクル／コマンド登録／メニュー登録×2／録音開始・停止の分岐／ミキサー状態／メーター中継／watcher 管理／終端イベント処理／embed 挿入／文字起こし起動×4／ミニ窓管理／remix。D の抽象化と、文字起こし系 4 メソッドの `transcribe/` への移設で 500 行程度までは自然に減る。Tier3 リファクタ（終端/起動処理の分割）は既に一度通っており、続きをやる素地はある。

### G. 配布整合性（半分解決済み）

「main.js だけ更新・バイナリ据え置き」は版数ハンドシェイク（0.6.1）で**検出**はできるようになった。残る改善余地: doctor の NG を待たず、**プラグイン起動時に abi 不一致を検知したら自動で「sysrec を取得」を提案（または自動更新）**する導線。BRAT 更新のたびに手動でバイナリも更新する現運用は事故が再発する。

### H. 小粒の負債（気づいた順・網羅ではない）

- [main.ts:434](src/main.ts#L434) 付近の stale コメント「Windows の per-source 分離は後続」— 実際は 0.5.0 で実装済み。仕様理解を誤らせる。

- `diagnostics.ts` 503 行: チェックが線形に増える構造だが、1 チェック 1 関数の形は保てているので現状維持で可。

## 4. プラットフォーム別の所見

### 4.1 macOS — 強みと弱み

- 強み: 状態機械の堅牢性（ファイルシステムが真実の源・レンダラ死亡に耐える・E2E で保護）。DRM 対応済みタップ。normalize による保存時の最終保証。
- 弱み: **デバイスライフサイクル（§2.1・B）が唯一かつ最大の穴**。加えて sysrec の診断ログが薄く（今回の 12:39 のログは 1 行）、過去にあった `[diag]` 詳細ログはデバッグビルド限りで消えている。障害解析はログよりファイル解析に頼った。

### 4.2 Windows — 強みと弱み

- 強み: 起動検証（録音が実際に始まるまで JSON を書かない）・track ended の fail-safe・逐次追記（クラッシュしても直近まで残る）・無音ウォッチ・mp4 不可時の webm フォールバック（文字起こし側も webm 対応済み）。防御パターンは macOS より新しい分だけ厚い。
- 弱み: **一度も実機で通していない**（0.6.0 の無音対策・0.7.0 の正規化とも）。クラッシュ復元は原理的に不可（レンダラ内録音の宿命・文書化済み）。ノイズゲートが無い（Chromium 任せ）。

### 4.3 Windows 実機でしか確かめられない項目（優先順）

1. 出力デバイス切替時のループバック挙動（追従？ ended？ 無音継続？）→ §2.2
2. 0.7.0 の取り込み時正規化が実際の WASAPI 信号で -16 dBFS に収束するか
3. 0.6.0 の無音対策（GC 保持・resume・無音ウォッチ）が #4 報告環境で解決になっているか
4. `MediaRecorder.isTypeSupported("audio/mp4")` が Windows 版 Electron でも true か（macOS 版 Chromium 142 では true を確認。ただし AAC は OS コーデック依存の可能性があり、false でも webm フォールバックで動く設計）
5. 手動ミキサーのゲイン反映とソース別メーター

`dev/win-audio-diag.js` は 1〜3 の切り分け用に既に用意されている。「Obsidian のコンソールに貼る → JSON が出る → それを貼り返してもらう」の 1 往復で取れる形に整備すると検証コストが最小になる。

### 4.4 参考: WebCodecs の制約（0.7.0 で確定した事実）

Chromium 142 の `AudioEncoder` は **AAC エンコード不可**（`mp4a.40.2: false` / `opus: true`）。一方 **MediaRecorder は AAC エンコード可**（プラットフォームコーデック使用）。ただし MediaRecorder はリアルタイム MediaStream しか受けないため「録音後のファイルを読み直して正規化して書き戻す」には録音実時間ぶんの再生が必要になり、保存後正規化は実用にならない。**取り込み時正規化（0.7.0 の方式）が Windows では構造的に正しい選択**である——この制約ごと設計書 §4.5 に記録済み。

## 6. やらない方がよいこと

- **録音エンジンの統一**（Windows も外部バイナリ化／macOS も Web Audio 化）: 前者は Windows 用ネイティブ開発の負担が大きく、後者は macOS の「レンダラより長生き」堅牢性を捨てる。二重化は現実解であり、解くべきは D（抽象の欠如）であってエンジン統一ではない。
- **SessionMeta の discriminated union 化**（前回調査から継続保留）: 型は美しくなるが録音状態モデル全域に触る割に、実際のバグはこの層で起きていない。
- **UI 層の大規模リファクタ**: RecordingView 487 行・controlWindow 209 行は大きいが、バグ源になっていない。ヘッドレス検証もできないので費用対効果が低い。
- **設定の再編・doctor の再設計**: 可読性の問題に留まり、実害ゼロ。

## 7. まとめ

「バグが多発」の実態は、**堅牢に作った状態機械の外側——音声 DSP の二重実装とデバイスとの接点——に検証の光が当たっていなかった**こと。UI や録音ライフサイクルは安定している。従って大規模な作り直しは不要で、①実害中の R0 を直し、②DSP 仕様を単一源化し（R1）、③今回手動でやった実音声検証を常設化する（R2）——この 3 点で「多発」の構造は断てる。R3 以降は再発防止の恒久化であり、段階的に進めればよい。
